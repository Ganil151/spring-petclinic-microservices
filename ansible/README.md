## Ansible Configuration
This directory contains Ansible playbooks and configuration files used for automating the deployment and management of infrastructure and applications.

### Structure
- `playbooks/`: Contains Ansible playbooks for various tasks. 
- `roles/`: Contains Ansible roles that can be reused across different playbooks.
- `inventory/`: Contains inventory files defining the hosts and groups of hosts.
- `ansible.cfg`: The main configuration file for Ansible settings.
- `requirements.yml`: Lists Ansible Galaxy roles and collections required for the playbooks.
- `README.md`: This file, providing an overview of the Ansible setup.
- `scripts/`: Contains helper scripts for Ansible operations.
- `vars/`: Contains variable files used in playbooks and roles.
- `templates/`: Contains Jinja2 templates used for configuration files.
- `files/`: Contains static files to be copied to remote hosts.
- `logs/`: Directory for storing logs generated by Ansible runs.
- `tests/`: Contains test playbooks and configurations for validating Ansible roles and playbooks.
- `docs/`: Additional documentation related to Ansible usage and best practices.
- `vault/`: Contains encrypted files using Ansible Vault for sensitive data.
- `modules/`: Custom Ansible modules developed for specific tasks.
- `plugins/`: Custom Ansible plugins for extending functionality.
- `callbacks/`: Custom callback plugins for logging and notifications.
- `filter_plugins/`: Custom filter plugins for data manipulation in playbooks.
- `library/`: Custom libraries used by Ansible modules and plugins.
- `meta/`: Metadata files for roles and playbooks.
- `handlers/`: Contains handler files for roles and playbooks.
- `defaults/`: Contains default variable files for roles.
- `tasks/`: Contains task files for roles.

### Usage

#### Inventory File
The `inventory.ini` file defines the hosts and groups of hosts that Ansible will manage.
```yaml
/etc/ansible/inventory.ini
# from the terminal run: 
sudo bash -c 'cat > /etc/ansible/inventory.ini << "EOF"
---
[mysql]
mysql-server ansible_host=54.167.194.172 ansible_user=ec2-user ansible_ssh_private_key_file=/home/ec2-user/.ssh/master_keys.pem

[all:vars]
ansible_python_interpreter=/usr/bin/python3
EOF'
```
---

#### Host File
The `hosts` file can also be used to define the same information as the inventory file.
```yaml
/etc/ansible/hosts.ini
# inside
# Run this command to recreate the file with zero hidden characters:
sudo tee /etc/ansible/inventory.ini >/dev/null << 'EOF'
[mysql]
mysql-server ansible_host=54.167.194.172 ansible_user=ec2-user ansible_ssh_private_key_file=/home/ec2-user/.ssh/master_keys.pem
EOF
```

#### Group Variables
The `group_vars/mysql.yml` file contains variables specific to the MySQL group.
```yaml
/etc/ansible/group_vars/mysql.yml
# inside
mysql_root_password: "petclinic"
mysql_petclinic_password: "PetClinic@12345!"

petclinic_databases:
  - customers
  - visits
  - vets

petclinic_users:
  - name: petclinic_user
    password: "{{ mysql_petclinic_password }}"
    priv: "*.*:ALL"
```
---

#### Main Playbook
The `mysql_setup.yml` playbook sets up the MySQL server for the Spring Petclinic application.
```yaml
/etc/ansible/mysql_setup.yml
# inside
---
- name: Configure MySQL for Spring Petclinic Microservices
  hosts: mysql
  become: true

  roles:
    - mysql_petclinic
```
---

Or use for `configure_mysql_server.yml`:
- Ensure MySQL server is installed, configured, and the Petclinic database and user are created.
```yaml
/etc/ansible/configure_mysql_server.yml
# inside
---
- name: Setup MySQL for Spring Petclinic
  hosts: mysql
  become: yes
  vars:
    mysql_root_password: "petclinic"
    petclinic_db: "petclinic"
    petclinic_user: "petclinic"
    petclinic_password: "PetClinic@12345!"
    mysql_port: 3306
  tasks:

    - name: Install MySQL server
      ansible.builtin.yum:
        name: mysql-server
        state: present
      when: ansible_os_family == "RedHat"

    - name: Install MySQL server (Debian/Ubuntu)
      ansible.builtin.apt:
        name: mysql-server
        state: present
        update_cache: yes
      when: ansible_os_family == "Debian"

    - name: Ensure MySQL service is running
      ansible.builtin.service:
        name: mysqld
        state: started
        enabled: yes
      when: ansible_os_family == "RedHat"

    - name: Ensure MySQL service is running (Debian/Ubuntu)
      ansible.builtin.service:
        name: mysql
        state: started
        enabled: yes
      when: ansible_os_family == "Debian"

    - name: Set root password
      ansible.builtin.mysql_user:
        login_user: root
        login_password: ''
        name: root
        password: "{{ mysql_root_password }}"
        host_all: yes
        priv: "*.*:ALL,GRANT"
      ignore_errors: yes

    - name: Remove anonymous MySQL users
      ansible.builtin.mysql_user:
        login_user: root
        login_password: "{{ mysql_root_password }}"
        name: ''
        host_all: yes
        state: absent

    - name: Remove test database
      ansible.builtin.mysql_db:
        login_user: root
        login_password: "{{ mysql_root_password }}"
        name: test
        state: absent

    - name: Create PetClinic database
      ansible.builtin.mysql_db:
        login_user: root
        login_password: "{{ mysql_root_password }}"
        name: "{{ petclinic_db }}"
        state: present
        encoding: utf8mb4

    - name: Create PetClinic user
      ansible.builtin.mysql_user:
        login_user: root
        login_password: "{{ mysql_root_password }}"
        name: "{{ petclinic_user }}"
        password: "{{ petclinic_password }}"
        host: "%"
        priv: "{{ petclinic_db }}.*:ALL"
        state: present

    - name: Apply schema and seed data
      ansible.builtin.mysql_db:
        login_user: "{{ petclinic_user }}"
        login_password: "{{ petclinic_password }}"
        name: "{{ petclinic_db }}"
        state: import
        target: /path/to/schema.sql
      # Repeat for data.sql if needed
```

#### Role Tasks
The `tasks/main.yml` file in the `mysql_petclinic` role includes other task files to create databases, users, and load schemas and data.
```yaml
/etc/ansible/roles/mysql_petclinic/tasks/main.yml
# inside
---
- name: Copy SQL Schema files to remote host for execution
  copy:
    src: "{{ role_path }}/files/{{ item }}-schema.sql"
    dest: "/tmp/petclinic/{{ item }}-schema.sql"
    owner: root
    group: root
    mode: '0644'
  loop:
    - customers
    - visits
    - vets
```
---

#### Creating Databases
The `create_databases.yml` task file creates the necessary databases for the Petclinic application.
```yaml
/etc/ansible/roles/mysql_petclinic/tasks/create_databases.yml
# inside
---
- name: Ensure MySQL databases exist
  mysql_db:
    name: "{{ item }}"
    state: present
    login_user: root
    login_password: "{{ mysql_root_password }}"
  loop: "{{ petclinic_databases }}"
```
---

#### Creating Users
The `create_users.yml` task file creates the necessary users for the Petclinic application.
```yaml
/etc/ansible/roles/mysql_petclinic/tasks/create_users.yml
# inside
---
- name: Create Petclinic MySQL users
  mysql_user:
    name: "{{ item.name }}"
    password: "{{ item.password }}"
    priv: "{{ item.priv }}"
    state: present
    login_user: root
    login_password: "{{ mysql_root_password }}"
  loop: "{{ petclinic_users }}"
```
---

#### Loading Schemas
The `load_schema.yml` task file loads the database schemas for the Petclinic application.
```yaml
/etc/ansible/roles/mysql_petclinic/tasks/load_schema.yml
# inside
---
- name: Load database schema
  mysql_db:
    name: "{{ item }}"
    state: import
    target: "/etc/ansible/petclinic/{{ item }}-schema.sql"
    login_user: root
    login_password: "{{ mysql_root_password }}"
  loop: "{{ petclinic_databases }}"
```
---

#### Loading Data
The `load_data.yml` task file loads the initial data into the Petclinic databases.
```yaml
/etc/ansible/roles/mysql_petclinic/tasks/load_data.yml
# inside
---
- name: Load sample data
  mysql_db:
    name: "{{ item }}"
    state: import
    target: "/etc/ansible/petclinic/{{ item }}-data.sql"
    login_user: root
    login_password: "{{ mysql_root_password }}"
  loop: "{{ petclinic_databases }}"
```

```bash
   ├── visits-schema.sql
```

### Running the Playbook
To run the playbook and set up the MySQL server, use the following command:
```bash
cd /etc/ansible
ansible-playbook -i inventory.ini mysql_setup.yml
This command will execute the playbook and set up the MySQL server for the Spring Petclinic application.
```
Add to Jenkins Pipeline (optional)

In your Jenkinsfile:
```groovy
stage('Setup MySQL Server') {
    steps {
        sh 'ansible-playbook -i /etc/ansible/inventory.ini /etc/ansible/mysql_setup.yml'
    }
}
This stage will run the Ansible playbook as part of your CI/CD pipeline to ensure the MySQL server is configured correctly before deploying the Spring Petclinic application.
```

Run:
```bash
cat -A /etc/ansible/inventory.ini
```



### Install PyMySQL on the MySQL server

```bash
sudo dnf install -y python3-pip
sudo pip3 install PyMySQL
```
Verify:
```bash
python3 -c "import pymysql; print('OK')"
```
If you see `OK`, PyMySQL is installed correctly.

### Fix inside Ansible
Update your Ansible playbook to ensure the required MySQL Python module is installed on the MySQL server before any MySQL tasks are executed.
```yaml
- name: Install MySQL Python dependencies
  yum:
    name: python3-PyMySQL
    state: present
```
Or use Pip:
```yaml
- name: Install PyMySQL via pip
  pip:
    name: PyMySQL
    executable: pip3
```
Place this task at the beginning of your playbook, before any MySQL-related tasks.
This ensures that the necessary Python module is available for Ansible to interact with the MySQL server, preventing the "No module named 'MySQLdb'" error.
### IMPORTANT — Also add this to your inventory
```yaml
[mysql]
mysql-server ansible_host=54.x.x.x ansible_user=ec2-user ansible_ssh_private_key_file=/home/ec2-user/.ssh/master_keys.pem

[all:vars]
ansible_python_interpreter=/usr/bin/python3
```
After that, rerun:
```bash
ansible-playbook -i inventory.ini mysql_setup.yml
```

Loads my_playbook.yml from the current working directory and:
 -i - uses my_inventory_file in the path provided for inventory to match the pattern.

 -u - connects over SSH as my_connection_user.

 -k - asks for password which is then provided to SSH authentication.

 -f - allocates 3 forks.

 -T - sets a 30-second timeout.

 -t - runs only tasks marked with the tag my_tag.

-M - loads local modules from /path/to/my/modules.

 -b - executes with elevated privileges (uses become).
-K - prompts the user for the become password.
